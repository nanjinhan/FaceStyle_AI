## 1. 이 앱이 뭔가

**친구들이 단체사진 한 장에 동시 접속해서 각자 자기 얼굴만 보정하는 앱.**

기존에는 사진을 보정 → 저장 → 카톡 전송 → 친구가 또 보정 → 저장을 반복하느라
JPEG 재압축이 쌓여 화질이 계속 나빠졌다. 이 앱은 **픽셀을 주고받지 않는다.**
서버에는 원본 1장만 있고, 오가는 건 "밝기 +12, 턱 -5" 같은 파라미터 JSON뿐이다.
최종 저장할 때 원본 해상도에 딱 한 번 렌더링하므로 **몇 번을 고쳐도 화질 손실이 0**이다.

- 상세 기능: [기능명세서.md](기능명세서.md)
- 설계·프로토콜: [아키텍처.md](아키텍처.md)
- 마일스톤 전체: [로드맵.md](로드맵.md)
- 출시 준비물(스토어·키·비용): [출시-준비물.md](출시-준비물.md)

**확정된 방향**: 단체사진 전용. 얼굴 1개당 1명 클레임, 본인 영역만 편집.
셀카(얼굴 1개) 여럿이 함께 보정하는 시나리오는 지원하지 않는다.

## 2. 지금 상태 한눈에

| | 상태 |
|---|---|
| 백엔드 (FastAPI) | 실시간 방 **완성** + 얼굴검출(YuNet) + 로그인/고유색/방생성. 앨범·알림은 미착수 |
| 앱 (Flutter) | 실시간 방/에디터 **완성** + **로그인·홈(방 생성)·마이페이지·초대참여 동작**. 앨범·알림·카메라는 껍데기 |
| 테스트 | 백엔드 63개, 앱 27개 — 전부 통과 |
| E2E 검증 | ✅ 로그인→토큰복원→방생성(얼굴검출)→호스트/게스트 join까지 실제 서버로 확인. 웹 빌드 컴파일 성공 |
| 브라우저 클릭 검증 | ⚠️ 코드·API·E2E는 검증됨. 크롬 2개로 사람이 클릭하며 보는 최종 확인은 미실시(아래 4장 ②) |

### 실제로 되는 것 (M1 = 코어 데모 완성)

- 방 만들기(사진 여러 장), 초대 링크/코드로 참여, 게스트 참여
- **업로드 시 서버가 얼굴 자동 검출**(YuNet) → 클레임 대상 생성
- 여러 명 동시 접속 → 한 사람이 슬라이더 움직이면 **다른 사람 화면에 즉시 반영**
- 슬라이더 6종(밝기/대비/채도/색온도/하이라이트/그림자)이 **사진에 실제로 적용됨**
- **얼굴 탭 → "이게 나예요" 클레임**, 내 얼굴=색 테두리, 남의 얼굴=잠금 아이콘
- **얼굴별 보정 슬라이더 11종**(피부/턱선/눈/코/입 등) — 내 얼굴만
- **실시간 커서** — 다른 참여자 마우스가 색상 점 + 닉네임으로 표시
- 내 얼굴만 편집 가능, 남의 얼굴 건드리면 서버가 거부 + 이유 안내
- 실행취소가 **내 작업만** 되돌림, 항목별 리셋, 전원 완료 시 자동 확정 + 편집 잠금

### 얼굴 보정(워핑) 현황 — 실제 픽셀 변형 동작

- 서버가 YuNet 5점 랜드마크(양눈·코·입양끝) + 이미지 크기를 저장 → 앱에 전달
- 앱: 삼각형 메시(`editor/rendering/face_warp.dart`)로 눈 크기·입술·코·얼굴축소를
  **실제로 변형**. PNG(`app/build/face_warp_preview.png`)로 눈 확대/축소 검증 완료
- 웹 브라우저에서 로그인→방생성→얼굴검출→클레임→편집까지 흐름 확인
- ⚠️ **주의**: 사진 업로드 시 `width/height`를 저장해야 얼굴 상자·워핑이 좌표를 잡는다
  (이걸 안 해서 얼굴 상자가 안 뜨던 버그를 잡음 — 옛 방의 사진은 크기 0이라 재업로드 필요)
- **실제로 되는 얼굴 보정**: 눈 크기·입술·코·얼굴축소(메시 워핑) + 피부 매끈·잡티(블러 블렌드)
- 남은 정밀화: 턱선·광대는 5점만으론 근사 → M4에서 MediaPipe 468점.
  피부는 정식 주파수 분리·눈입 마스킹이 M4 (지금은 블러 블렌드 근사)

### 디자인

- 브랜드: 보라→핑크 그라데이션(`core/theme/brand.dart`), 둥근 모서리 테마
- 스플래시(초기 로딩)·로그인·홈에 브랜드 적용. 나머지 화면은 기본 테마만 입혀진 상태

### 아직 안 되는 것

- **실제 소셜 로그인**(카카오/구글) — 지금은 닉네임만으로 로그인하는 개발용 스텁.
  키 준비되면 버튼만 붙이면 됨(백엔드 social-login은 그대로 씀). [출시-준비물.md](출시-준비물.md) 참고
- 앨범 알림, 인앱 카메라 — 아직 껍데기
  (앨범 상태 뱃지·할 일 필터·방장 마감은 ✅ 완료: 34411de)
- 회원 탈퇴 — 앨범 위임/삭제 로직이 필요해 이후
- 딥링크(`facestyle://join`) OS 연동 — 지금은 앱 안에서 초대코드 입력으로 참여
- 턱선·광대 정밀 워핑, 피부 주파수 분리 — M4 (MediaPipe 468점)
- 모바일 갤러리 저장 — 웹 다운로드는 됨, 모바일은 플러그인 연동 필요(M4)

### 앨범 사진 편집 — ✅ 됨 (M3 2차, cdf79a5)

앨범 사진을 탭하면 실시간 방과 **같은 에디터**로 들어가 얼굴 클레임·워핑·피부·저장·완료확정을
할 수 있다. 구현: 앨범 사진마다 **장수명 EditSession**을 하나 만들어 편집 인프라를 그대로 재사용
(`POST /albums/{id}/photos/{photoId}/edit-session` → sessionId/photoId/memberToken).
문서에 원래 적었던 "Photo.session_id nullable" 대신, 세션을 실제로 붙이는 방식이라 편집 코드
재작성이 전혀 없었다. 여러 앨범 멤버가 같은 사진 세션에 각자 다른 시간에 참여 = 비동기 협업.

정밀 랜드마크 검출은 M4에서 온디바이스 MediaPipe로 확장 예정(아래 "M4 보정 고도화 방향").

**테스트 사진** (`backend/test_photos/`, git 미추적): 이미지 예시1.jpg(5명)/예시2.jpg(2명)/예시3.jpg(3명).
백엔드 허용 형식은 `.jpg/.jpeg/.png/.heic` — webp/avif는 업로드 거부되므로 jpg로 변환해 쓴다.

### ② 브라우저 클릭 검증 — ⚠️ 부분 완료

웹 빌드(`app/web/`)를 추가했고 컴파일·E2E(서버→업로드→검출→세션)까지 확인했다.
다만 **크롬 창 2개로 사람이 직접 클릭하며 "같이 보정"을 눈으로 보는 최종 확인은 아직**이다.
확인하려면:
```bash
# 터미널 1 — 백엔드
cd backend && .venv/Scripts/python.exe -m uvicorn app.main:app --host 0.0.0.0 --port 8000
# 터미널 2 — 웹 앱 (크롬)
cd app && flutter run -d chrome --dart-define=API_BASE_URL=http://localhost:8000
```
이제 앱만으로 된다: 닉네임 로그인 → 홈에서 "사진 올려서 방 만들기" → 방 입장.
다른 크롬 창에서 그 방의 초대코드로 게스트 입장하면 2인 동시 보정을 볼 수 있다.
(Android SDK 미설치라 에뮬레이터는 아직 못 씀 — 웹으로 대체 중.)

⚠️ 웹에서 사진 업로드가 되려면 백엔드를 `--host 0.0.0.0`로 띄우고 API_BASE_URL을
브라우저가 접근 가능한 주소(localhost)로 맞춰야 한다.

### ③ Flutter 버전 불일치

이 PC는 Flutter 3.41.9(Dart 3.11.5)인데 프로젝트는 원래 Dart ^3.12.2를 요구했다.
진행을 위해 `pubspec.yaml`을 `^3.11.0`으로 낮춘 상태(`5f072a2`).
팀에서 Flutter 버전을 맞추고 되돌리는 게 맞다.

### ④ 백엔드 스키마 변경 — 기존 로컬 DB 폐기 필요

`EditSession.global_edit_policy`, `Photo.finalized_at`, `PhotoCompletion` 테이블이 추가됐다.
SQLAlchemy `create_all`은 기존 테이블에 컬럼을 못 붙이므로,
**예전에 서버를 돌린 적 있으면 `backend/togethersnap.db`를 지우고 다시 켜야 한다.**
또 `POST /sessions`의 필드명이 `file` → `files`로 바뀌었다.

### M4 보정 고도화 방향 (2026-07-21 논의 확정)

SNOW/B612급 보정 품질의 핵심은 얼굴을 "네모 박스"가 아니라 "수백 개의 점"으로
아는 것이다. 상용 앱들과 같은 종류의 기술이 전부 무료·온디바이스로 공개돼 있다.

| 단계 | 기술 | 아는 것 | 가능한 보정 |
|---|---|---|---|
| 지금(M1) | OpenCV YuNet (서버) | 얼굴 위치 bbox | 클레임 구분만 — 보정용 아님 |
| M4 ① | MLKit | 눈코입 + 윤곽 133점 | 기본 워핑 |
| M4 ② | **MediaPipe Face Mesh** | **3D 468점** | 상용 앱급 워핑·피부 보정의 기반 |

M4에서 구현할 3계층 파이프라인 (전부 온디바이스, GPU 셰이더):
1. **얼굴 메시** — MediaPipe Face Mesh로 468점 추적
2. **워핑** — 메시 점을 밀어 형태 변형 (턱 깎기 = 턱선 점을 안쪽으로, 눈 키우기 = 눈 주변 점을 바깥으로). 파라미터(`faces.face_N.jawSlim` 등)는 이미 백엔드에 정의돼 있음
3. **피부 보정** — 주파수 분리(결/톤 분리 후 결만 스무딩) + 피부 세그멘테이션 마스크로
   눈썹·머리카락 보호. 단순 블러는 플라스틱 피부가 되므로 금지

이 방식은 우리 아키텍처(파라미터만 동기화, 렌더링은 각 기기)와 정확히 맞물린다 —
서버는 변경할 것이 없고, 각 기기가 같은 파라미터로 같은 결과를 렌더링한다.

## 6. 개발 환경 세팅

```bash
# 백엔드
cd backend
python -m venv .venv
.venv/Scripts/python.exe -m pip install -r requirements.txt   # Windows
.venv/Scripts/python.exe -m pip install pytest httpx           # 테스트용
.venv/Scripts/python.exe -m uvicorn app.main:app --reload      # http://localhost:8000
.venv/Scripts/python.exe -m pytest tests/ -q                   # 57개

# 앱
cd app
flutter pub get
flutter analyze
flutter test                                                   # 28개
flutter run --dart-define=API_BASE_URL=http://10.0.2.2:8000    # 안드로이드 에뮬레이터
```

### 크롬으로 웹 테스트 (Android SDK 없이) — 삽질로 배운 주의점

1. **백엔드는 IPv4로**: `--host 0.0.0.0`. `--host ::`(IPv6)로 띄우면 브라우저의
   127.0.0.1 접속이 안 된다. 그리고 앱 주소는 `localhost`가 아니라 **`127.0.0.1`**을 써야 한다
   (브라우저가 localhost를 IPv6로 풀어버려 IPv4 백엔드에 못 붙는 문제).
2. **릴리스 빌드 + 캐시 끄기**: 디버그 웹은 흰 화면이 오래 간다. 릴리스로 빌드해 서빙하고,
   캐시 때문에 옛 빌드가 뜨면 `Cache-Control: no-store` 헤더를 주는 서버로 서빙한다.
   ```bash
   flutter build web --release --dart-define=API_BASE_URL=http://127.0.0.1:8000
   # build/web 을 no-cache 헤더로 서빙 (python http.server 커스텀), 접속: 127.0.0.1:5000
   ```
3. 방을 만들려면 로그인이 필요 → 앱에서 닉네임 로그인 → "사진 올려서 방 만들기".
   두 번째 크롬 창에서 초대코드로 게스트 입장하면 2인 동시 편집 확인 가능.
4. **얼굴 상자/워핑이 안 보이면** 먼저 사진의 width/height가 저장됐는지 의심
   (옛 방 사진은 크기 0 → 재업로드 필요).

API 주소 기본값은 `http://10.0.2.2:8000`(안드로이드 에뮬레이터에서 본 호스트 PC).
실기기면 PC의 LAN IP를 `--dart-define`으로 넘긴다.

## 7. 이어서 작업할 때 붙여넣을 프롬프트

새 대화창에서 아래를 그대로 붙여넣으면 된다.

```
FaceStyle 프로젝트를 이어서 작업할 거야.
docs/이어서-작업하기.md 를 먼저 읽고, 이어서 docs/기능명세서.md,
docs/아키텍처.md, docs/로드맵.md 를 확인해줘.

그 다음 실제 코드와 문서가 일치하는지 확인하고,
- 지금 어디까지 돼 있는지
- 다음에 뭘 해야 하는지
- 시작 전에 내가 정해줘야 할 게 있는지
정리해서 알려줘. 그리고 진행하자.
```

특정 작업만 시키고 싶으면 뒤에 덧붙인다:

```
이번엔 얼굴 워핑에 피부 스무딩(잡티/피부 매끈)을 추가하자.
```

### 다음에 이어서 할 만한 것 (얼굴 보정 계속)

- **피부 스무딩** — 잡티 제거·피부 매끈. 주파수 분리(결/톤) + 얼굴 영역 마스크
- **턱선·광대 정밀화** — 5점 → MediaPipe 468점(온디바이스)으로 윤곽 확보
- **로컬 웹 실행 방법**은 아래 6장 참고 (백엔드 IPv4 + 앱 API_BASE_URL=127.0.0.1)

### 작업할 때 지켜온 규칙

- **명세서가 기준.** 코드와 명세가 다르면 명세를 따르되, 어긋난 지점은 문서에 기록한다
- API/프로토콜을 바꾸면 [아키텍처.md](아키텍처.md)에 **먼저** 반영하고 구현한다
- 권한·동시편집처럼 조용히 깨지는 기능은 **반드시 테스트를 붙인다**
  (특히 "남의 얼굴 못 건드림" 같은 건 공격 시나리오로 검증)
- 작업이 끝나면 이 문서의 2·3·5장과 헤더 날짜/커밋을 갱신하고 커밋한다
