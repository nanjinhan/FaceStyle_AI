# 아키텍처

## 전체 구성

```
┌─────────────────────────────┐        HTTPS (REST)        ┌──────────────────────────────┐
│  Flutter 앱 (iOS/Android)   │ ◄────────────────────────► │  FastAPI 백엔드 (backend/)   │
│                             │        WSS (실시간)        │                              │
│  - 화면/상태: Riverpod      │ ◄────────────────────────► │  - REST: 인증/앨범/방/사진   │
│  - 얼굴인식: MLKit(온디바이스)│                           │  - WS: 실시간 방 동기화      │
│  - 보정 렌더링: 온디바이스   │      FCM 푸시 알림         │  - DB: SQLite → PostgreSQL   │
│    (이미지 필터 파이프라인)  │ ◄──────────────────────────│  - 파일: 로컬 → S3 호환      │
└─────────────────────────────┘                            └──────────────────────────────┘
```

핵심 설계 원칙 (친구 백엔드 v0.1에서 그대로 계승):

1. **보정 = 파라미터 편집.** 픽셀을 주고받지 않는다. 서버는 사진 원본 1장 + 보정 파라미터 JSON(전역/얼굴별)만 관리하고, 각 클라이언트가 파라미터를 받아 온디바이스로 렌더링한다. 실시간 동기화 트래픽이 KB 단위라 6인 방에서도 렉이 없다.
2. **LWW(Last-Write-Wins) + 단조 증가 버전.** 같은 파라미터를 동시에 만지면 마지막 쓰기가 이긴다. 클레임 기반 영역 분리 덕에 실제 충돌은 공용 영역에서만 발생 가능하고, 공용 영역은 기본 방장 전용이라 충돌 확률이 낮다.
3. **내 것만 편집.** 얼굴 클레임(얼굴 1개당 1명) → 클레임한 사람만 그 얼굴의 파라미터 편집 가능. 이 권한 검사를 **서버(WS 핸들러)에서 강제**한다 (현재 v0.1은 클라이언트 신뢰 — 보강 필요, 로드맵 M1 참고).

## 데이터 모델

### 현재 있는 것 (backend/app/models.py)

`User`, `EditSession`(실시간 방), `Member`(방 참여자), `Photo`, `Face`(클레임 포함),
`EditStateRecord`(보정 파라미터), `EditHistory`, `ExportResult`

### 추가해야 하는 것 (명세 v1.0)

| 모델 | 용도 | 명세 근거 |
|---|---|---|
| `User.color` | 가입 시 자동 배정되는 고유 색상 (hex) | 회원관리 — 고유 색상 배정 |
| `Album` | 그룹 앨범 (이름, 방장, 초대 토큰) | 앨범 — 생성 |
| `AlbumMember` | 앨범 멤버십 (role: owner/member, 알림 on/off) | 앨범 — 멤버/알림 |
| `AlbumPhoto` | 앨범 사진 (업로더, 상태: waiting/editing/done, 확정 여부) | 앨범 — 업로드/조회/확정 |
| `PhotoCompletion` | 사진×멤버 완료 체크 (전원 완료 시 자동 확정) | 실시간 방·앨범 — 완료 확정 |
| `Notification` | 알림 (종류, 대상 앨범/사진, 읽음 여부) | 메인 — 알림 목록 |
| `DeviceToken` | FCM 푸시 토큰 | 앨범 — 업로드 알림 |
| `EditHistory` 활용 | member_id별 undo 스택 분리 | 실시간 방 — 개인별 실행취소 |

실시간 방과 앨범 비동기 보정은 **같은 편집 모델(EditStateRecord + 클레임)을 공유**한다.
방이 24시간 후 만료되면 사진·편집 상태를 앨범으로 이관한다 (방 종료 → 앨범 보관 정책).

## 실시간 프로토콜 (WS)

`WS /ws/sessions/{session_id}?token={memberToken}` — 친구 v0.1 프로토콜을 유지하고 확장한다.

| 방향 | 타입 | 상태 | 설명 |
|---|---|---|---|
| C→S | `edit` / `undo` / `redo` | 있음 | undo/redo를 **member_id 기준 개인 스택**으로 변경 예정 |
| C→S | `presence` | 있음 | 커서 좌표(`cursor: {x,y}`) 필드 추가 예정 → 색상 커서 표시 |
| C→S | `lock_param` / `unlock_param` | 있음 | 소프트 락 (그대로 사용) |
| C→S | `reaction` | 있음 | 이모지 리액션 |
| C→S | `reset_param` | 추가 | 기능별 리셋 (본인 영역 항목만) |
| C→S | `complete` / `uncomplete` | 추가 | 완료 확정 체크 (전원 완료 시 서버가 `finalized` 브로드캐스트) |
| S→C | `state_sync`, `edit_applied`, `presence_update`, `param_locked/unlocked`, `face_claimed/released`, `member_kicked`, `export_ready`, `reaction` | 있음 | |
| S→C | `completion_update`, `finalized` | 추가 | 완료 현황 / 최종 확정 + 편집 잠금 |

**권한 강제(서버측)**: `edit`/`reset_param` 처리 시 `path`가 `faces.{faceId}.*`이면
그 얼굴의 `claimed_by_member_id == member.id`인지 검사, `global.*`이면 방 설정
(방장 전용/전체 허용)을 검사. 위반 시 무시하고 에러 응답.

## 얼굴 인식·보정 렌더링

- **얼굴 검출**: 클라이언트 온디바이스가 기본 (`google_mlkit_face_detection`). 업로드 시 bbox를 서버에 등록. 서버 OpenCV 검출(`face_detection.py`)은 보조/선택.
- **보정 렌더링**: 파라미터 → 이미지 변환은 전부 온디바이스. 밝기/대비 등 전역 보정은 ColorMatrix, 피부/윤곽 등 얼굴 보정은 얼굴 랜드마크 기반 워핑·스무딩. 저장 시 원본 해상도로 재렌더링 후 갤러리에 저장.
- 전원 확정 시의 "최종본"은 확정 시점의 파라미터 스냅샷이며, 각 기기가 동일 파라미터로 렌더링한다 (렌더링 코드가 같으므로 결과 동일).

## 인증·초대

- 유저 토큰(JWT, `typ: user`)과 방 멤버 토큰(`typ: member`) 2단 구조 — v0.1 그대로.
- 게스트: 초대 링크 → 닉네임만으로 member 토큰 발급 (이미 구현됨). 게스트 데이터는 기기 로컬 보관.
- 딥링크: `facestyle://join?token=...` (+ Android App Links / iOS Universal Links). 미설치 시 스토어로 보내고, 설치 후 최초 실행 때 pending 초대 토큰으로 자동 입장 (deferred deep link).

|---|---|---|
