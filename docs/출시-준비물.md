# 출시 준비물 체크리스트

> 앱스토어/플레이스토어 실제 출시 기준. 개발자(Claude)가 대신 할 수 없고 **계정 소유자 본인만** 할 수 있는 것들을 정리한다.

## 0. 한 줄 요약

**API 사용료는 전부 0원이다.** 돈이 드는 건 스토어 등록비(구글 $25 1회 / 애플 $99 매년)와 서버 호스팅뿐이며, 서버도 무료 티어로 시작할 수 있다.

## 1. 왜 "앱 키"가 필요한가

명세서 1장(회원관리)에 소셜 로그인이 있다. 유저가 "카카오로 로그인" 버튼을 누르면 카카오 서버 입장에서는 *"이 요청을 보낸 앱이 대체 어떤 앱인가"* 를 알아야 한다. 그걸 증명하는 신분증이 앱 키다.

- **데이터를 사는 게 아니다.** 각 개발자 사이트에서 무료로 발급받는다.
- 발급 자체는 앱당 5~10분.
- 키는 앱 소유자 계정에 묶이므로 반드시 본인이 발급해야 한다.

## 2. 이 앱에는 "외부 데이터 API"가 없다

흔한 오해라 짚어둔다. 날씨 앱처럼 남의 데이터를 사오는 구조가 아니다.

| 구성요소 | 어디서 오나 | 월 비용 |
|---|---|---|
| 사진 | 유저가 직접 올림 | 0원 |
| 보정 파라미터 | 우리 백엔드(FastAPI) | 서버비만 |
| 얼굴 인식 | 구글 MLKit — **폰 안에서 실행**, 오프라인 동작 | 0원 |
| 실시간 동기화 | 우리 백엔드 WebSocket | 서버비만 |
| 푸시 알림 | Firebase Cloud Messaging | 0원 (무제한) |

보정이 "픽셀이 아니라 파라미터(KB 단위)만 주고받는" 구조라(→ [아키텍처.md](아키텍처.md) 설계 원칙 1) 서버 부하가 매우 가볍다. 그래서 초기 서버비도 거의 안 든다.

## 3. 단계별 준비물

### 지금 (1~4단계, 로컬 개발) — 준비물 없음, 0원

- [ ] **테스트용 사진 2~3장**
  - ⚠️ 단체사진(얼굴 2~4명)이 필요한지는 **제품 방향 확정 후 결정** (아래 6장 참고)
- 그 외 키·계정 일절 불필요. 전부 로컬에서 돌아간다.

### M2 (회원가입·로그인·초대) — 전부 무료 발급

- [ ] **카카오 앱 키** — https://developers.kakao.com → 내 애플리케이션 → 앱 만들기
  - 필요한 것: 네이티브 앱 키, REST API 키
  - 카카오 로그인 활성화 + 리다이렉트 URI 등록 필요
- [ ] **Firebase 프로젝트** — https://console.firebase.google.com
  - 여기서 Google 로그인 OAuth 클라이언트가 자동 생성된다 (따로 만들 필요 없음)
  - 안드로이드 앱 등록 → `google-services.json` 다운로드 → 나에게 전달
  - iOS도 낸다면 → `GoogleService-Info.plist`도 함께
  - 요금제: **Spark(무료)** 로 충분
- [ ] (선택) **도메인** — 딥링크를 `https://` 형태로 예쁘게 쓰려면 필요
  - 없어도 `facestyle://join?token=...` 커스텀 스킴만으로 동작한다
  - 연 1~2만원 수준

### M3 (푸시 알림) — 무료

- [ ] Firebase 프로젝트에 FCM 포함되어 있음 (M2에서 이미 준비 완료)
- 발송량 **무제한 무료**

### 출시 시점 — 여기서만 실제 비용 발생

- [ ] **Google Play 개발자 등록 — $25 (1회, 평생)**
  - https://play.google.com/console
  - 본인 확인용 신분증 + 결제수단 필요, 승인까지 며칠 걸릴 수 있음
- [ ] **Apple Developer Program — $99 (매년)**
  - iOS 출시 / Apple 로그인 / Universal Links 전부 이게 있어야 가능
  - ⚠️ **애플 심사 규정**: 소셜 로그인이 있는 앱은 **Apple 로그인도 반드시 제공**해야 한다. iOS에 카카오 로그인을 넣으려면 Apple 로그인도 같이 구현해야 함.
- [ ] **백엔드 서버 호스팅**
  - Fly.io / Railway / Render 무료·소액 티어로 시작 가능
  - 사진 저장: 초기엔 서버 디스크 → 유저 늘면 Cloudflare R2(10GB 무료)로 이전
- [ ] **개인정보처리방침 URL** (스토어 등록 시 필수 입력)
  - 사진·얼굴 데이터를 다루므로 특히 중요. 무료 생성기 사용 가능.

## 5. 준비 순서 (권장)

1. **지금** — 테스트 사진만 준비 → 1단계(보정 렌더링) 개발 시작
2. **이번 주 중** — 카카오 개발자 가입 + Firebase 프로젝트 생성 (각 10분, 무료)
   → M2 작업 시작할 때 키만 꽂으면 바로 진행됨
3. **출시 직전** — Google Play $25 결제 + 개인정보처리방침 준비
4. **나중에 (선택)** — Apple $99/년

## 6. 제품 방향 (확정, 2026-07-20)

**명세서 그대로 간다.** 단체사진 한 장에 여러 명이 **동시 접속해서 각자 자기 얼굴을 보정**하는 앱.

### 해결하려는 문제

기존: 사진 보정 → 저장 → 카톡 전송 → 친구가 다시 보정 → 저장 → 다시 전송…
- JPEG 재압축이 누적되어 **화질이 계속 나빠진다**
- 누가 뭘 고쳤는지 추적 불가, 최신본이 뭔지 헷갈림
- 순서대로 한 명씩 해야 해서 느림

해결: [아키텍처.md](아키텍처.md) 설계 원칙 1 — **픽셀을 주고받지 않는다.**
- 서버에는 원본 1장만 존재. 오가는 건 "밝기 +12, 턱 -5" 같은 파라미터 JSON뿐
- 최종 저장 시 원본 해상도에 **단 한 번** 렌더링 → **재압축 손실 0**
- 트래픽이 KB 단위라 6인 동시 접속도 안 끊김

### 확정된 상호작용 규칙

| 규칙 | 명세 항목 | 구현 상태 |
|---|---|---|
| 각자 자기 얼굴만 편집, 타인 얼굴은 잠금 | 편집 권한 분리 | ❌ B3 — 2단계에서 서버 강제 |
| 한 명이 만지는 항목은 잠금 표시 | 소프트락 | ✅ 구현됨 (`lock_param`) |
| 누가 어디를 터치 중인지 색상 커서로 실시간 표시 | 실시간 커서 | ⚠️ 절반 — 아래 |
| 실행취소는 본인 작업만 | 개인별 실행취소 | ❌ B5 — 2단계 |
| 전원 완료 체크 시 최종본 확정 | 완료 확정 | ❌ B7 — 3단계 |

**셀카(얼굴 1개) 협업은 지원하지 않는다.** 얼굴 1개당 1명 클레임 원칙 유지. 필요해지면 나중에 예외 규칙으로 추가.

### 발견된 누락 항목: 커서 좌표

명세 "실시간 커서 — 각 참여자가 편집 중인 위치를 **색상 동그라미 커서**로 실시간 표시"가
로드맵 B목록에 빠져 있다. 현재 [ws.py](../backend/app/routers/ws.py) `presence`는 `tool`/`region`만
주고받고 `cursor: {x, y}` 좌표가 없다. → **2단계에 포함시켜 처리한다.**

- 4단계(클레임 UI)에서 "A가 B 얼굴을 못 건드린다"를 검증하려면 필수
